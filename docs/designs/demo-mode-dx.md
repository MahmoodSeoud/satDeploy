## Vision

### 10x Check
satdeploy as a platform you can evaluate without installing anything beyond Python and Docker.
`satdeploy demo start` pulls a pre-built GHCR image, starts a simulated satellite (zmqproxy + agent),
auto-configures the CLI, and prints a guided tutorial. A satellite engineer goes from `git clone`
to first successful deploy in under 3 minutes (first run, includes Docker pull). Subsequent runs
start in ~10 seconds. The demo *teaches* what satdeploy does by letting the user do it.

### Platonic Ideal
A satellite engineer finds satdeploy on GitHub, sees a terminal recording in the README,
runs two commands, and within minutes has deployed a binary to a simulated satellite, rolled it back,
and seen the version history. They think: "This is exactly what I've been building ad-hoc scripts for."
The tool earns trust by working flawlessly on first contact. No YAML editing, no port numbers, no
protocol knowledge required.

### EUREKA
Everyone assumes the DX problem is "make Docker easier." But the real problem is that satdeploy has
no demo target. Every deployment tool ships with a way to try it against something (Ansible has localhost,
Terraform has local providers, Mender has QEMU devices). satdeploy's Docker setup already IS a virtual
satellite. `satdeploy demo start` auto-configures for this simulator.

## Architecture

### Core Approach (C: Full Ideal Architecture)
```
$ git clone ... && cd satBuild
$ pip install -e .
$ satdeploy demo start
  -> checks Docker >= 20.10 with Compose v2 (docker compose), exits with helpful error if missing
  -> pulls ghcr.io/mahmoodseoud/satdeploy-sim:latest (multi-stage image, <300MB target)
  -> starts zmqproxy + agent containers via embedded docker-compose config
  -> waits for agent readiness: calls CSPTransport.get_status() directly (max 30s, 2s intervals)
  -> writes demo config to ~/.satdeploy-demo/config.yaml
  -> sets SATDEPLOY_CONFIG_DIR=~/.satdeploy-demo in a shell alias hint
  -> prints guided tutorial with copy-pasteable commands
$ satdeploy status --config-dir ~/.satdeploy-demo
$ satdeploy push test_app --config-dir ~/.satdeploy-demo
$ satdeploy rollback test_app --config-dir ~/.satdeploy-demo
$ satdeploy demo stop
```

### Demo Config (exact values)
```yaml
# ~/.satdeploy-demo/config.yaml
name: demo-satellite
transport: csp
zmq_endpoint: tcp://localhost:9600
agent_node: 5425
ground_node: 40
zmq_pub_port: 9600
zmq_sub_port: 9601
backup_dir: /opt/satdeploy/backups
max_backups: 5

apps:
  test_app:
    local: ~/.satdeploy-demo/binaries/test_app  # v2 copied here by demo start
    remote: /opt/demo/bin/test_app               # path inside container
    service: null
    param: null
```

The `demo start` command:
1. Creates `~/.satdeploy-demo/binaries/` and copies v2 there (the "new version" to deploy)
2. The agent container has `/opt/demo/bin/test_app` pre-populated with v1
3. When user runs `satdeploy push test_app`, v2 is uploaded to replace v1 on the container
4. No binary swap occurs — the local file stays as v2. The demo is one-shot: push v2, rollback to v1.
   Re-running `demo start` resets the container to v1 for a fresh demo.

### Demo Binary
```bash
#!/bin/sh
# demo/v1/test_app — pre-installed on simulated satellite
echo "test_app v1.0.0 (demo)"

#!/bin/sh
# demo/v2/test_app — what the user deploys
echo "test_app v2.0.0 (demo)"
```
Stored in `demo/` directory in the repo. `demo start` copies v2 to `~/.satdeploy-demo/binaries/test_app`
as the "new version" to deploy. The container starts with v1 pre-installed at `/opt/demo/bin/test_app`.

### Agent Readiness Probe
The `demo start` command polls readiness by instantiating `CSPTransport` directly:
```python
t = CSPTransport(zmq_endpoint=..., agent_node=5425, ground_node=40, ...)
t.connect()
try:
    result = t.get_status()
    if result:  # non-empty dict = agent is ready
        return True
except TransportError:
    pass  # not ready yet
finally:
    t.disconnect()
```
IMPORTANT: Must call `connect()` before `get_status()` — without it, `get_status()` silently
returns `{}` (no socket initialized), which is indistinguishable from "not ready."

Retry logic:
- Max attempts: 15
- Interval: 2 seconds
- Total timeout: 30 seconds
- On timeout: print error with `docker compose logs agent` output for debugging

No new health endpoint needed — reuses existing STATUS command.

### `satdeploy demo stop` Behavior
- Runs `docker compose -f <demo-compose> down` to stop and remove containers
- Does NOT delete `~/.satdeploy-demo/` (preserves config for next `demo start`)
- `satdeploy demo stop --clean` removes `~/.satdeploy-demo/` entirely

### `satdeploy demo status` Behavior
- Checks if demo containers are running (`docker compose ps`)
- If running: prints agent CSP address, zmqproxy ports, config path
- If not running: prints "Demo environment is not running. Start with: satdeploy demo start"

### `satdeploy demo eject` Behavior
- Prompts: "What transport will you use? (ssh/csp)"
- Generates `~/.satdeploy/config.yaml` with:
  - All fields populated with commented placeholders
  - Transport-specific fields pre-filled based on choice
  - Comments explaining each field in satellite context
- Prints: "Config template written to ~/.satdeploy/config.yaml — edit it for your hardware."
- Example output for CSP:
```yaml
name: my-satellite        # Name for this target (e.g., "flatsat-1", "em-2")
transport: csp
zmq_endpoint: tcp://localhost:9600  # ZMQ proxy address (change to your proxy host)
agent_node: 5425                    # CSP node address of the satellite agent
ground_node: 40                     # CSP node address of your ground station
# ...
```

### Docker Image Strategy
The GHCR image uses a multi-stage Dockerfile to minimize size:
```
Stage 1 (builder): Ubuntu 24.04 + full build toolchain
  - COPY repo source (including git submodules: libcsp, libparam, DTP in satdeploy-agent/lib/)
  - Clones CSH (public repo), builds it
  - Builds satdeploy-agent from source (baked in, not volume-mounted)
  - Builds zmqproxy
  NOTE: Submodules (libcsp, libparam, DTP) live in satdeploy-agent/lib/.
  CI pipeline: use `actions/checkout@v4` with `submodules: true`.
  Local builds: user must have run `git submodule update --init --recursive`.
  Dockerfile.sim should verify submodule dirs are non-empty and fail early if not.

Stage 2 (runtime): debian-slim
  - Copies only: satdeploy-agent binary, zmqproxy binary, libcsp.so, libzmq.so
  - Copies demo/v1/test_app to /opt/demo/bin/test_app (with execute bit set)
  - Creates /opt/satdeploy/backups/ (agent backup directory)
  - Includes /bin/sh (standard in debian-slim, used by demo scripts)
  - Target size: <300MB (debian-slim base ~80MB + runtime libs)
```

**Repo-public requirement:** GHCR free tier requires public repos for public package visibility.
The repo MUST be public before the GHCR pipeline works for anonymous pulls. If the repo is still
private, `demo start` falls back to local `docker compose build` (slower but functional).
The fallback logic:
1. Try `docker pull ghcr.io/mahmoodseoud/satdeploy-sim:latest`
2. If pull fails (404 or network error), warn user: "Pre-built image not available. Building
   locally — this may take ~5 minutes on first run."
3. Fall back to `docker compose build` using the repo's Dockerfile.sim
4. The "subsequent runs start in ~10 seconds" claim applies only when the GHCR image is cached.

### Idempotency
- If demo containers are already running, `demo start` prints "Demo already running" and
  re-prints the tutorial. Does not restart containers.
- To force a fresh start: `satdeploy demo stop && satdeploy demo start`

### Docker Compose for Demo
`demo start` uses an embedded docker-compose YAML string constant in `demo.py` (not the repo's
`docker-compose.yml` which is for development). Written to `~/.satdeploy-demo/docker-compose.yml`
on every run (always overwrites to pick up version upgrades). This demo-specific compose config:
- Uses the GHCR image (not `build: .`)
- Exposes ports 9600/9601
- Starts zmqproxy and agent only (no build service, no csh service)
- Pre-populates `/opt/demo/bin/test_app` with v1

### SATDEPLOY_CONFIG_DIR Environment Variable
**Required change:** Add `envvar="SATDEPLOY_CONFIG_DIR"` to the `--config-dir` Click option in
`cli.py`. This is a prerequisite for the tutorial UX (avoids `--config-dir` on every command).
Currently, `DEFAULT_CONFIG_DIR` in `config.py` is hardcoded to `~/.satdeploy`. The env var
overrides this at the CLI level.

## Reviewer Concerns

Issues surfaced across 3 rounds of adversarial review that require attention during implementation:

1. **Agent node address must be verified at C level.** The demo uses `agent_node: 5425` but the
   CLAUDE.md example shows `5424`. Verify the compiled-in default in `satdeploy-agent` source
   before hardcoding in the demo config.

2. **`zmq_endpoint` port semantics are misleading.** `CSPTransport` ignores the port in
   `zmq_endpoint` (uses `zmq_pub_port`/`zmq_sub_port` instead). The eject template comment
   "ZMQ proxy address (change to your proxy host)" implies the port matters. Clarify or fix
   during implementation.

3. **Demo re-entrancy after push+rollback.** The demo is one-shot (push v2, rollback to v1).
   A second `push` after rollback pushes v2 again (same hash), which may confuse users.
   Document this limitation in the tutorial or handle it.

4. **`demo eject` is the lowest-priority accepted scope item.** If timeline is tight, implement
   it as a separate commit that can be deferred without blocking the core demo flow.
