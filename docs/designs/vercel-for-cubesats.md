## Vision

### 10x Check
Imagine a YC partner in Q4 2026 watching the demo. They say "wait, nobody has this?" The answer isn't faster iterate — it's **fleet**. The moment you're managing 3+ flatsats or a constellation simultaneously, single-target tooling feels like an escalator. Every commercial sat operator is or wants to be a constellation. Fleet primitives shipped in Phase 0 make Phase 1 pitch 2× stronger.

### Platonic Ideal
**Developer POV:** `satdeploy iterate` → breakpoint in 8s. `satdeploy watch` → save file → auto-iterate → logs stream. Every error ships its fix command. Shareable URLs per change. Multi-target feels identical to single-target. First-time setup is 60s.

**Customer POV:** CTO demos to board in 3 min. Dashboard has exec view. Pilot-to-paid is inevitable because daily hook is installed.

**Startup POV:** YC partner reaction "nobody has this?" Cold-email conversion measurable. Self-serve trial path.

## Accepted Scope (Phase 0 — rescoped for ~10 week calendar, DX-revised 2026-04-18)

**Binary size assumption:** 500KB controller binary (typical DISCO-2 payload). Transport: ZMQ/TCP for dev (p50 ≤10s achievable); fast CAN for integration tests. All performance claims scoped to these configs.

### Core wedge — `satdeploy iterate <app>` + `satdeploy watch`
- **`satdeploy iterate <app>`** — edit-to-running in p50 ≤10s on flatsat dev configs (ZMQ/TCP), p95 ≤30s
- **`satdeploy watch <app>`** — file watcher (watchdog) triggers iterate on save. The actual daily-loop hook. Typing `iterate` 40×/day is itself DX friction; `watch` is the retention magic. **In Phase 0 per DX review 2026-04-18 (swapped with `history` pretty-printer).**
- **Acceptance:** iterate exits 0 when service is running with new binary; exits non-zero with typed error otherwise. ≥95% success rate across 100 iterations on DISCO-2. watch loops indefinitely; Ctrl+C exits cleanly; prints iterate latency per cycle.
- **Shell autocomplete for app names** — Click-native `satdeploy iterate <TAB>` completes from config. ~1h setup. Stripe/Fly parity.
- **Sysroot sync** — two paths:
  - **Prebuilt path (DISCO-2):** on first `iterate` against a DISCO-2 flatsat, detect manifest hash, lazy-fetch `https://satdeploy.io/sysroots/<hash>.tar.zst` (or GitHub Release), cache to `~/.satdeploy/sysroots/<hash>/`. **Solves the 60-second install-to-iterate promise per DX review 2026-04-18.** One-time publish step as Week 2 deliverable.
  - **User-SDK path (other targets):** `SATDEPLOY_SDK` env var or `--sdk` flag. Fallback when manifest hash has no prebuilt. Clear error message: "No prebuilt sysroot for hash abc123. Set SATDEPLOY_SDK=/path/to/yocto-sdk or run `satdeploy sysroot publish` if you've built it."
- **ABI-compat check** — before upload, diff DT_NEEDED + symbol versions of local binary vs target sysroot. Fail loudly with "missing libparam.so.3 on target; run satdeploy sync-sysroot or install libparam."
- **Local debuginfod server** — `satdeploy debuginfod serve` wraps `elfutils` package's `debuginfod` binary. Indexes `~/.satdeploy/sysroots/<hash>/debug/`. Served at `http://localhost:8002`. Environment: `DEBUGINFOD_URLS=http://localhost:8002` set by `satdeploy gdb`.
- **Incremental diff upload — bsdiff locked.** Store last-deployed binary per app on target in `{backup_dir}/<app>/last`. Compute bsdiff patch locally, upload patch + apply via `bspatch` on target. Fallback to full upload if no `last` file.
- **Error→fix table (Week 2 deliverable)** — enumerated Python dict in `satdeploy/errors.py` mapping ~10-15 common errors to typed messages + suggested satdeploy commands. Covers: ABI mismatch, missing symbol, sysroot stale, service won't restart, backup corrupt, validate timeout, gate refused/override, debuginfod port busy, gdbserver missing, CSP disconnect, bsdiff compute fail, bspatch apply fail, manifest hash unknown, no prebuilt sysroot, rsync fail. Each entry: trigger, error text (libparam-example quality), suggested command with estimated time ("(12s)"). **Makes WOW #4 real per DX review 2026-04-18.**

### Core stick — flight safety (DX-revised: warn+override default)
- **`satdeploy validate`** — runs `validate_command` from config (default: `./tests/run.sh`). Passes env vars: `SATDEPLOY_APP`, `SATDEPLOY_HASH`, `SATDEPLOY_TARGET`. Records to history.db `validations` table: `(id, app, hash, outcome, started_at, finished_at, stdout_tail, stderr_tail)`. `outcome` ∈ {"pass", "fail", "error"}.
  - **Acceptance:** validate exits with code of wrapped command; writes exactly one validations row per invocation; tail truncated to 4KB each.
- **`satdeploy push <app>` — warn+override by default (DX-revised 2026-04-18)** — before upload, compute SHA256 of local binary. Query validations table for `(app=<app>, hash=<sha256>, outcome='pass')`.
  - **No match:** print warning "Hash abc123 has no PASS record. Proceed anyway? [y/N]" with one-line rationale. On `y`: deploy + log `[OVERRIDE]` row in history.db with user identity + timestamp. On `N` or Ctrl+C: abort with exit 2.
  - **Match:** deploy + log `[VALIDATED]` row.
  - **`--strict` flag or config `require_strict: true`:** hard-block on no-match with typed error, no prompt. For flight customers who want enforced gate.
  - **Acceptance:** Default interactive behavior never silently deploys unvalidated. `[OVERRIDE]` and `[VALIDATED]` entries always logged with user identity. `--strict` refuses with exit 2 on no-match. Hash-based lookup (staleness impossible).
  - **Why this flip:** Reviewer + design doc Open Q0 both identified hard-block as paternalistic for agile CubeSat teams. Audit trail (not refusal message) is the disaster-prevention story.
- **Rollback** — already exists. Phase 0 verifies p95 rollback ≤60s across 20 trials on DISCO-2.
- ~~**`satdeploy history <app>` pretty-printer**~~ — **moved to Phase 1** (DX-swap for `satdeploy watch`). history.db data already in dashboard; direct `sqlite3` queries work for thesis evidence.

## Implementation Landmines (eng review 2026-04-19 — mandatory)

From eng-focused outside voice subagent. 16 findings accepted. Each is concrete, not speculative. Implementer MUST address each before the relevant module ships.

### P0 — will brick flatsats or ship broken

1. **Watch cancel-anywhere bricks flatsat.** Cancelling mid-SFTP or mid-bspatch leaves torn file on target. FIX: agent writes to `<remote>.tmp`, atomic `rename()` to final path at the end. Watch cancel point is ONLY pre-rename. Post-rename, let it finish (or add explicit rollback).
2. **debuginfod PID file TOCTOU.** Two `satdeploy watch` terminals both see missing PID file, both spawn, second bind fails silently. FIX: `fcntl.flock` on `~/.satdeploy/run/debuginfod.pid.lock` during check-and-spawn. Verify PID via `/proc/<pid>/comm == "debuginfod"`. After kill+respawn on sysroot hash change: poll `/proc/<pid>` until gone, OR set `SO_REUSEADDR` on socket bind.
3. **validate process group kill.** `os.killpg(p.pid, SIGTERM)` only works if `pid==pgid` — requires `start_new_session=True` (py≥3.11) or `preexec_fn=os.setsid` on Popen. Send signal to `-pgid` (negative). SIGTERM first with 2-3s grace, then SIGKILL (`./tests/run.sh` may spawn Docker that ignores SIGTERM).
4. **bsdiff4 version skew.** `bsdiff4>=1.2.4` emits patches that older `bspatch` rejects. FIX: pin `bsdiff4==1.2.3` in pyproject.toml OR ship matched-source `bspatch` in satdeploy-agent build. Verify with Week 1 feasibility test: local patch, target apply, SHA256 byte-equal.

### P1 — correctness fixes that prevent subtle bugs

5. **bsdiff memory = 17× old size.** 20MB binary → 340MB RSS on dev laptop during compute. FIX: in `_compute_patch`, if `len(old_bytes) > 5 * 1024 * 1024`, skip bsdiff and return None (signal full upload).
6. **history.py migrate race.** Two processes both see missing `validations` table, both try to create. FIX: `CREATE TABLE IF NOT EXISTS validations (...)`. Also switch migration to `PRAGMA user_version` for schema versioning (currently uses column-existence checks which work but don't scale).
7. **history.py hardcoded `transport="ssh"` (line 146).** iterate/validate/gate silently log wrong transport. FIX: `record.transport or "ssh"` instead of literal.
8. **watchdog misses editor-rename saves.** VSCode/Vim/JetBrains atomic-rename; `FileModifiedEvent` alone misses them. FIX: register `FileCreatedEvent` + `FileMovedEvent` on parent directory, filter by basename match.
9. **Global debounce storms on package saves.** `git checkout` or "save all" → N files in 200ms → N iterates fire. FIX: debounce per-(app, dest-path) key. Coalesce by file key, not globally.
10. **uvicorn.run() in main thread freezes CLI.** `satdeploy dashboard` must spawn `uvicorn` as subprocess (`subprocess.Popen(["uvicorn", "satdeploy.dashboard:app", ...])`), not call `uvicorn.run()` inline. Cold-boot 400-800ms justifies the process fork.

### P2/P3 — polish + test infra

11. **Ctrl+C + watchdog Observer threading.** Non-daemon threads don't respond to SIGINT. FIX: main thread installs signal handler; handler calls `observer.stop()` + `observer.join()` + kills iterate subprocess group.
12. **sqlite3.connect per-request leaks fds in dashboard.** FIX: FastAPI `Depends()` with `yield conn; conn.close()` pattern. Thread-local connections, never module-level.
13. **HTMX + Jinja has no CSRF.** Fine for read-only Phase 0. **Phase 1 flag:** when "override" button ships, add `fastapi.middleware.cors.CORSMiddleware` restrictions + HMAC-signed action tokens.
14. **errors.py regex matching.** stderr strings are never exact matches. Store regex patterns keyed to stderr snippets. Integration test replays real failed-run stderr through matcher — matcher regressions are the most common "we shipped wrong error message" bug.
15. **macOS FSEventsObserver coalesces differently than Linux inotify.** Pin `watchdog>=3.0`. In CI, force `PollingObserver` for deterministic test timing: `from watchdog.observers.polling import PollingObserver` in tests.
16. **Test mocking path.** Patch at `satdeploy.validate.subprocess.Popen`, NOT top-level `subprocess.Popen`. Tests pass locally with direct patching but break in CI where import order differs. Same pattern for watchdog: drive events directly via `handler.on_modified(FileModifiedEvent(...))`, not sleep-and-pray.

## R6 design spec (plan-design-review 2026-04-20)

Design decisions locked before R6 implementation. Initial score 3/10 → 9/10 after
resolving 5 unresolved decisions.

### Information hierarchy (scan-test priority)

```
┌──────────────────────────────────────────────────────────────┐
│ STATUS BAR (full-width, green/yellow/red per existing tiles) │
│  ● deployed • ⟲ rolled_back • ✕ deploy_failed                │
│   controller @ som1 — 2026-04-20 09:15:08Z                   │
│   hash abc123 · by kat (cli) · git:3f4a2e1                   │
├──────────────────────────────────────────────────────────────┤
│  [AUDIT TIMELINE]  dense monospace, one-line-per-event       │
│    2026-04-20 09:15:00Z  iterated     kat      500KB, 2.1KB  │
│    2026-04-20 09:15:03Z  validated    ci       PASS (3s)     │
│    2026-04-20 09:15:08Z  deployed     kat      [VALIDATED]   │
│                                                              │
│  [DIFF]            git-show output, Jinja `| e`, <pre>       │
│                                                              │
│  [VALIDATE OUTPUT] <details> collapsed by default            │
│    stdout/stderr tails (4KB each)                            │
├──────────────────────────────────────────────────────────────┤
│                               [ Rollback to abc123 ]         │
└──────────────────────────────────────────────────────────────┘
```

Priority: status (did it succeed?) → timeline (what/who/when) → diff + validate
(why) → actions (separated, bottom). Forensic density, not alert-pretty.

### State table

| State | What user sees |
|---|---|
| Happy path | Green bar. Full page as above. Rollback enabled. |
| No git_hash | `[DIFF]` replaced: "No git provenance — binary deployed via `--local` without commit reference." No fix CTA. |
| Commit not in local clone | `[DIFF]` shows: "Commit `<sha>` not in local git. Run `git fetch origin` or view externally." + `[copy hash]` button. |
| No validate result | `[VALIDATE]` replaced: "No validation run. Deploy used `--strict` fail or `[OVERRIDE]`." Link to `[OVERRIDE]` entry. |
| Rollback already performed | Yellow banner: "⟲ Rolled back 2h ago. Current `som1` runs → `/iterations/def456`." Rollback disabled. |
| Rollback failed | Red banner: "Last rollback failed: `<typed_error>`." Retry button + link to logs. Rollback enabled (retry = same button). |
| 404 unknown hash | Clean 404: "No iteration found. Hash may have been pruned by `max_backups` rotation or URL malformed." Back link → `/`. |

### Rollback confirmation UX

**Type-to-confirm modal.** Click "Rollback" → modal with text input labeled
"Type `rollback som1 abc123` to confirm". Submit button disabled until exact
match. Cancel button + Esc close modal. High-friction matches destructive-on-flight-software
stakes. Decision per plan-design-review 2026-04-20 tradeoff vs two-step-button.

### Permalink durability

URL `/iterations/<hash>` is stable for lifetime of history.db row. Degradations
specified in the state table above cover: purged commit, dashboard-vs-git-host
mismatch, renamed target/app, dashboard not reachable cross-network. Phase 1
cloud tier removes last limitation.

### Audit timeline rendering

Real `<table>` with `<th scope="col">`. Monospace font (inherits ticker font from
live-activity ticker). ISO 8601 timestamps. One row per event. Event types:
`iterated`, `validated`, `deployed`, `rolled_back`. No icons, no gradients, no
row separators beyond single-pixel border. Mission engineers scan this like log
output.

### Responsive + a11y

Desktop/tablet: full hierarchy. Mobile (<768px): status bar stacks, timeline
becomes cards, diff collapsed by default. Keyboard tab order: timeline → diff
expand → validate expand → rollback. Status bar `role="status"`. Rollback button
≥44px tall mobile. Status colors WCAG AA 4.5:1 minimum.
